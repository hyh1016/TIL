
## Read/Write Concern
- 다중 노드 기반의 분산 시스템 환경에서 데이터의 일관성(정합성), 내구성, 응답 속도 간 Trade-Off를 조율하기 위한 설정
- Read Concern: 노드가 어떤 방식으로 데이터를 제공할 것인가를 결정하며, 이 옵션에 따라 얼마나 '확실한' 데이터를 읽을지, 얼마나 '빠르게' 읽을지가 결정됨

## Read Concern
- 얼마나 확실한 데이터를 얼마나 빨리 읽을 것인가? (일관성 vs 응답 속도)

| **옵션**                | **설명**                          | **특징 및 권장 용도**                                      |
| --------------------- | ------------------------------- | --------------------------------------------------- |
| **`local`** (default) | 해당 노드가 가진 최신 데이터를 즉시 반환         | 지연 시간은 가장 짧지만, 롤백될 가능성이 있는 데이터를 읽을 수 있음             |
| **`available`**       | 샤딩(Sharding) 환경 최적화             | `local`과 비슷하지만, 분산 환경에서 더 낮은 레이턴시를 목표로 함. 롤백 가능성 존재 |
| **`majority`**        | 과반수 노드가 인정한 **확정된 데이터**만 반환     | 읽기 시 다른 노드의 상태를 체크하므로 약간 느릴 수 있지만, 롤백 걱정이 없음        |
| **`linearizable`**    | **가장 강력한 일관성**. 읽기 시점의 최신 상태 보장 | 읽기 직전에 Primary가 여전히 유효한지 확인 통신을 수행. 성능 비용이 가장 큼     |
| **`snapshot`**        | **트랜잭션용**. 특정 시점의 데이터 뷰 제공      | 다중 문서(Multi-document) 트랜잭션에서 원자적이고 일관된 읽기를 위해 사용    |

## Write Concern
- 몇 대의 노드에 쓰기가 완료되어야 애플리케이션에 성공 응답을 보낼 것인가? (내구성 vs 응답 속도)

| **옵션**                        | **설명**                                               | **특징 및 권장 용도**                                         |
| ----------------------------- | ---------------------------------------------------- | ------------------------------------------------------ |
| **`w: 0`**                    | **Unacknowledged**. 쓰기 요청 후 응답을 기다리지 않음              | 네트워크 오류만 감지 가능. 유실되어도 무방한 대량의 로그 데이터 수집 시 사용           |
| **`w: 1`**                    | **Acknowledge (Default)**. Primary 노드의 메모리에만 기록되면 성공 | 가장 일반적이지만, Primary 장애 시 데이터 유실(Rollback) 위험이 있음        |
| **`w: <number>`**             | 지정된 **n개**의 노드에 복제 완료 시 성공                           | 복제 수동 제어 필요 시 사용 (예: 3대 노드 중 2대 보장)                    |
| **`w: "majority"`** (default) | **과반수** 노드에 복제 완료 시 성공                               | **가장 권장됨**. 장애 시에도 데이터 유실이 없음을 보장 (High Availability). |
| **`w: <tag set>`**            | 사용자가 정의한 **태그 세트**에 기반한 복제                           | 물리적으로 다른 데이터 센터 간 복제 보장 등 복잡한 토폴로지 관리용                 |
| **`j: true`**                 | **Journaling**. 디스크의 저널 로그에 기록된 것까지 확인               | 정전 등으로 노드가 비정상 종료되어도 복구 가능하도록 내구성 극대화                  |
| **`wtimeout`**                | 쓰기 응답 대기 **제한 시간 (ms)**                              | 네트워크 장애 등으로 인해 애플리케이션 쓰기가 무한 대기하는 것을 방지                |

## 사용 사례
- 쓰기량이 많은데 유실을 어느 정도 허용할 수 있는 경우 성능 향상을 위해 w:1 옵션을 쓰기도 함
- 데이터의 정합성이 중요한 로직(ex: 금융)은 w: "majority"와 readConcern: "majority"를 주로 사용
- readPreference: "secondary" 옵션을 쓰는 경우 readConcern: "majority" 옵션을 사용해야 Stale Read 현상을 예방할 수 있음
