
## MongoDB의 기본 Primary Key, ObjectID
### 구조
![](imgs/Pasted%20image%2020260419104939.png)
- 12 bytes 문자열
- 의미 없는 난수 문자열이 아니라, 위와 같은 구조로 구성됨
	- **Timestamp: 생성 시간 (4 bytes)**
		- 현재 날짜 기준 값으로, 날짜 범위의 정렬과 탐색을 가능하게 만듦
	- **Random Vaule: 무작위 값 (5 bytes)**
		- 말 그대로 무작위로 부여되는 값이다. 겹칠 확률은 1조분의 1 보다도 낮음 (2^40)
	- **Counter: 증가 카운터 (3 bytes)**
		- 노드 별로 랜덤으로 부여된 후 단조 증가한다. Timestamp가 초 단위까지만 기록하므로 동일 애플리케이션 서버 내에서 밀리초 단위로 아이디를 생성할 때의 충돌을 막기 위한 값
### 왜 Auto Increment  키를 사용하지 않을까?
- RDB와의 설계 철학 차이. RDB는 중앙 집중식 단일 서버를, MongoDB는 **분산 환경의 다중 서버**를 전제로 제작됨
- 분산 환경에서 RDB 처럼 단조 증가하는 키를 사용하게 되면 아래와 같은 이슈가 발생
	- 각 노드에서 공유할 수 있는 전역 카운터가 필요해짐
	- 이를 위해서는 모든 노드가 자신의 키 카운터 상태를 상시 동기화하거나, 키를 중앙 집중식으로 관리하는 노드를 두어야 함
	- 이는 복잡도를 높이고 성능 병목을 초래함
	- 따라서, 노드 별로 동기화하지 않아도 중복될 일이 없도록 정교하게 설계된 값을 사용하는 것임 (0%는 아니지만 매우 미미함)
- 또한 Object ID 방식을 사용하게 되면 **클라이언트 사이드(애플리케이션 서버 내 Mongo Driver)에서 ID를 만들 수 있음**
	- MongoDB 서버에서 아이디를 생성하게 되면 DB가 연산을 위해 리소스를 사용한다는 점, 생성한 아이디를 획득하기 위한 클라이언트-서버 간 통신이 발생한다는 단점이 있음
	- 따라서, 각 MongoDB 드라이버에서 엄격한 BSON 표준에 따라 아이디를 생성 및 사용하도록 함 (모든 공식 드라이버는 동일 구현 방식을 준수하므로 언어가 달라져도 영향받지 않음. 하지만 비공식 서드파티 라이브러리 라면 영향이 있을 수도?)
### 재밌는 사실 - 과거에는 Random Vaule 구성 방식이 달랐음
- 현재는 Random Value 부분은 말그대로 무작위 값으로 구성되는데, 과거에는 머신 ID 3 bytes와 프로세스 ID 2 bytes로 구성되었음
- 왜 이렇게 바뀌었을까? 이 방식이 클라우드와 컨테이너 기술(Docker & Kubernetes)이 자리잡으며 문제를 야기했기 때문
- 클라우드와 컨테이너 기술 환경에서는 서로 다른 애플리케이션이어도 동일한 머신에서 동작하는 경우가 많음. 또한, 도커 컨테이너 내부에서 실행되는 메인 애플리케이션은 거의 첫 번째 프로세스로서 PID 1을 부여받는 경우가 많음
- 이렇게 되면 오히려 머신 ID와 프로세스 ID가 동일할 확률이 높아져 충돌 확률이 올라가고 마는 것
- 따라서 무작위 값으로 변경되었으며, 가상화 및 서버리스 환경으로의 진화에 잘 대응한 스펙 변경이라고 볼 수 있음!

## ObjectID vs Snowflake ID
- 이 내용을 공부하면서 가장 많이 떠오른 지식은 바로 Snowflake ID
- 두 기술 모두 "분산 환경에서 중앙 서버 병목 없이 고유한 ID를 어떻게 만들 것인가?" 라는 난제에 대한 해결안
- 아래와 같은 차이점이 있음

|항목|ObjectID|Snowflake ID|
|---|---|---|
|크기|96 bits|**64 bits** ✅|
|타입|hex 문자열|**정수(Long)** ✅|
|시간 정밀도|초|**밀리초** ✅|
|단조 증가|❌ 미보장|**✅ 보장**|
|생성 주체|클라이언트|서버 노드|
|인프라 조정|**불필요** ✅|워커 ID 사전 할당 필요|
|최대 노드 수|**사실상 무제한** ✅|1,024개|
|용도|MongoDB 전용|**범용 (RDB 등)** ✅|
- 그렇다면 Snowflake ID가 더 정교하고 더 사이즈도 작아보이는데 MongoDB는 왜 ObjectID 방식을 쓸까?
- Snowflake는 인프라 복잡도를 감수하더라도 최소한의 용량, 최대한의 정밀도를 보장하고자 하는 기술
	- Snowflake ID 방식은 아이디를 전역 관리할 중앙 코디네이터가 필요
	- 분산 시스템에서는 서버 간 클럭 불일치로 인한 충돌 문제가 잦아서, 밀리초 단위의 아이디는 관리하기 어려울 수 있음
	- 워커 ID의 사전 할당이 필요하므로 노드 구성에 잦은 변경이 있다면 관리가 어려움
- 그에 반해 ObjectID는 단순함을 통한 관리 용이성도 챙기되 충돌을 막음
	- 중앙 관리자 없이 클라이언트가 ID를 생성할 수 있다는 편리함
	- 워커 노드의 개수에 변화가 있어도 별도 조치가 필요 없음
	- 서버 간 시계가 미세하게 어긋나더라도 ID 생성에 문제가 없으며, 각 노드가 1초 당 약 160만개 (2^24) 정도의 고유 아이디를 만들어낼 수 있음
- 정리하자면, 각 기술은 목적과 그에 따른 Trade-off가 조금씩 다르다는 것!
