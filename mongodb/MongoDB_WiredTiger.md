
## WiredTiger
- **MongoDB의 기본 스토리지 엔진**
- 스토리지 엔진이란?
	- 데이터베이스 관리 시스템(DBMS)에서 데이터를 실제로 디스크나 메모리에 저장, 조회, 업데이트, 삭제(CRUD)하는 핵심 컴포넌트

## 특징
### Document 단위의 동시성 제어
- 초기 MongoDB는 데이터베이스/컬렉션 단위로 락을 관리해 동시 처리에 한계가 있었으나 WiredTiger는 Document 단위로 락을 관리하여 병렬성 향상 (여러 작업이 동일 Document를 수정하려 할 때에만 충돌이 발생)
### MVCC와 스냅샷
- MVCC(다중 버전 일관성 제어)를 통해 데이터 일관성을 유지
- 쓰기 작업 수행 시 스냅샷을 생성해 다른 읽기들이 기존 데이터를 읽을 수 있도록 함
### 내부 캐시와 메모리 관리
- WiredTiger는 자체적인 '내부 캐시(Internal Cache)'와 운영체제의 '파일 시스템 캐시(OS Cache)'를 모두 활용
- 캐시 사이즈는 `50% of (RAM - 1GB)` 또는 `256MB` 중 더 큰 값을 사용
- 나머지 메모리는 OS가 파일 시스템 캐시로 사용하여 압축된 데이터를 메모리에 유지하는 데에 사용
### 데이터 압축
- 데이터를 디스크에 쓸 때 압축을 지원하여 스토리지 비용, 디스크 I/O 비용을 절감
- 컬렉션 데이터는 **Snappy** 압축 알고리즘을 기본으로 사용하고, 인덱스는 **Prefix(접두사)** 압축을 사용
	- 스토리지 공간이 극도로 부족하다면 압축률이 더 높은 `zlib`이나 `zstd`으로 변경할 수도 있으나 CPU 사용량이 증가하는 Trade-off가 있기 때문에 병목 지점을 정확히 파악하고 결정해야 함 (CPU인가, 디스크 I/O와 저장공간 부족인가)
### Checkpoint와 Journaling
- MongoDB는 데이터를 일정 주기마다 디스크에 백업하는 동작을 수행. 이게 Checkpoint
- 그러나 백업 후 다음 백업 이전에 DB가 다운되면 그 사이의 데이터는 Checkpoint에 반영되지 않는데, 이 데이터를 복구하기 위해 사용하는 기술이 바로 Journaling
	- 모든 쓰기가 기록되는 저널 파일을 통해 실행했던 쓰기들을 재실행해 복구하는 방법
		- 모든 쓰기가 기록된다고 하니 로그 크기가 무한정 커질 것 같지만, 체크포인트 이전 시점의 기록은 알아서 날린다.
- Write concern 옵션을 조정해 저널 파일(디스크) 기록 후 ACK vs 메모리에만 저장 후 ACK를 제어할 수 있음
