
## 1. 읽기 성능 최적화
- 읽기 요청이 많은 서비스의 읽기 성능 최적화를 위해 Redis를 사용할 수 있음
- 일반적으로 **Look-Aside (Cache-Aside) + Write-Around** 전략을 사용
	- Look-Aside (Cache-Aside): Redis에서 먼저 읽고 없으면 RDB에서 읽음 (읽고 Redis에 저장)
	- Write-Around: 모든 쓰기는 RDBMS가 처리하며, 쓰기 처리 후 Redis 캐시를 갱신
- **실시간성이 중요한 데이터에는 이 전략을 사용하면 안 됨**
	- 왜? DB를 수정한 뒤 이 변경이 즉각 레디스에 반영된다고 100% 보장할 수 없기 때문
		- case 1: DB 수정 후 기존 Redis 캐시를 삭제하기 전에 애플리케이션이 죽을 수도 있음
		- case 2: 읽기 발생(A) -> 데이터 없어 DB에서 조회(A) -> 다른 스레드가 쓰기 수행(B) -> 캐시 삭제 시도했으나 캐시가 없어서 아무 것도 삭제되지 않음(B) -> old 데이터 읽은 뒤 캐시에 저장(A) 순서로 이루어지면 캐시에 new 데이터가 아닌 old 데이터가 저장됨
	- 따라서, 실시간성 까지는 필요 없는 데이터여도 캐시 데이터에 TTL을 설정해 특정 시간 이후에는 데이터가 갱신될 수 있도록 해야 함
- **캐시 스탬피드**에 주의
	- 캐시 스탬피드 현상: 조회가 매우 많은 캐시가 만료되며 DB에 순간적으로 트래픽이 몰려 DB가 다운되는 현상
	- 이를 예방하기 위해 모든 캐시에 동일 TTL을 적용해 일괄 만료하기 보다는 Jitter 기반의 TTL을 설정하는 것이 좋음
- 읽기 요청이 매우 많은 시스템이라면 **Write-Through** 방식을 사용하는 방법이 있음
	- Write-Through: 캐시와 RDB에 쓰는 것을 동기로 수행
	- 데이터 변경 시에도 캐시를 invalidate 하지 않고 항상 캐시에서 읽어갈 수 있음 (Always warm-up)
		- 다만 TTL 만료에 의해서는 여전히 Cache-miss가 발생할 수 있음
		- 그리고 시스템이 복잡해지고, 트랜잭션 관리가 복잡해짐
	- 이 방식을 사용할 시, 반드시 RDB에 먼저 쓰고 Redis에 써야 함. 안 그러면 Redis에 썼는데 DB에 쓰기 전에 죽어버리면 사용자에게는 실패로 보고되었으나 데이터 읽기 시에는 데이터가 보이는 문제가 생기며(Phantom data) Redis TTL 만료 시에는 또 이 데이터가 사라지는 문제가 발생

## 2. 쓰기 성능 최적화
- 선착순 이벤트(쿠폰 발급, 티켓팅 등)와 같은 폭발적 쓰기 트래픽 대응을 위해서도 Redis를 사용할 수 있음
- Write-Behind (Write-Back) + Message Queue 방식을 사용
	- Write-Behind (Write-Back) + Message Queue: 캐시에 쓰고 응답한 뒤 비동기로 RDB에 반영
		- 쓰기 처리량 최적화가 가능
- Redis 재고 차감 후 사용자에게는 성공으로 응답했으나 이후 단계 미실행으로 실제 데이터는 유실될 위험이 있음
	- 이 경우 재고는 차감되지만 사용자에게는 제공되지 않는 이슈가 발생
	- 하지만 선착순 이벤트처럼 재고가 정해져있는 경우 차라리 '미달 발급'을 택하는 게 좋으므로 이를 의도된 설계로 가져가는 것도 괜찮음
- Redis 자체가 죽으면 어떻게 해야 할까?
	- 1000개 중 500개 발행 후 Redis Down (RDB에서도 500개 차감)
	- 만약 RDB/AOF 기반의 Redis 자동 복구 전략을 사용하면 (뒤의 몇 초 변경이 날라가서) 500개 보다 많은 재고로 회복될 수 있음 (초과 발행! 매우 큰 문제!)
	- 따라서, RDBMS(최종 진실의 출처) 개수 기반으로 수동 회복 해줘야 함

## Redis의 데이터를 100% 신뢰 가능한 데이터로 취급하지 말라
- 위의 모든 과정을 보면, 모두 'Redis가 죽었을 때 데이터의 기준은 RDBMS를 참고하는' 방식의 설계를 사용
- Redis는 인메모리 DB이고, 성능을 최적화 하기 위해 백업 데이터를 항상 최신화 하지 않음 (물론 이렇게 운용하는 옵션도 있지만 성능이 확 저하되므로 거의 안 씀)
- 그러니 Redis 데이터를 100% 신뢰하는 방향으로 설계를 하면 안 된다. Redis가 죽었을 때 데이터가 유실되는 문제가 발생하므로
- 가장 효율적이며 의도된 Redis의 사용처는 '캐시 스토리지'라는 사실

## 이렇게까지 하면서 Redis를 써야 하나?를 항상 고민하라
- 높은 처리량을 가정하며 1번과 2번 설계를 항상 적용하는 것은 대부분의 상황에서 '오버 엔지니어링'
- RDBMS는 생각보다 빠르고 강하다.
- 그러니 목표 TPS를 산정하고, RDBMS만으로도 이 TPS가 처리 가능한지 테스트 해보고, 불가능하다고 판단될 때에만 Redis를 도입해 최적화를 진행하는 것이 더 현명하다.
