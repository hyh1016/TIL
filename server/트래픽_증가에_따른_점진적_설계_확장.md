
- 가장 단순한 형태의 아키텍처에서, 트래픽이 증가함에 따라 더 많은 트래픽을 효율적으로 처리하기 위해 아키텍처가 진화하는 과정을 흐름에 따라 설명
- 설명은 AWS 리소스를 기준으로 하나 다른 public cloud를 사용한다 해도 골자는 비슷함

## 가장 간단한 설계: 단일 서버에서 모든 것을 처리
![](imgs/Pasted%20image%2020260409220830.png)
- 서버 한 대에 FE, BE, DB, 정적 파일 등 모든 것을 위치시키는 구성
- 하나의 서버만 관리하면 되기 때문에 간단하고 단순하며, 저비용
- 모든 프로그램이 컴퓨팅 자원을 공유하므로 금방 성능 한계에 부딪힘
- 특히 데이터베이스가 꽤 많은 컴퓨팅 자원(CPU, Memory, Disk)을 사용하기 때문에 DB가 금방 병목이 됨

## 데이터베이스를 분리
![](imgs/Pasted%20image%2020260409221020.png)
- 병목이 되는 요소인 데이터베이스를 별도 서버로 분리한 구성

## 정적 파일 제공 최적화 (정적 파일 제공하는 경우)
### 정적 파일 서버를 분리
![](imgs/Pasted%20image%2020260409221249.png)
- 트래픽이 많아지면 정적 파일(이미지, 영상, html 등)을 제공하는 과정이 병목이 될 수 있음 (제공하는 정적 파일이 있다는 가정 하에)
- 정적 파일을 서빙하는 별도 스토리지 서비스(S3)를 구축한 구성. 애플리케이션 서버는 (용량이 커서 Memory, Network 전송 리소스를 더 많이 요구하는) 정적 파일 서빙으로부터 자유로워짐
- 하지만 위에서 말한 것처럼 정적 파일은 용량이 큰 경우가 많아서, 파일을 요청한 클라이언트와 정적 파일 제공 서비스 간 거리가 멀수록 응답 속도가 느려짐
### CDN을 적용
![](imgs/Pasted%20image%2020260409222156.png)
- CDN을 적용하면 물리적 거리로 인해 증가하는 네트워크 Latency를 최적화할 수 있음
- 또한, CDN이 없다면 Origin 서버로 집중되는 트래픽을 지역별 별도 서버로 분산할 수 있다는 장점도 있음

## 애플리케이션 서버 Scale-Out
![](imgs/Pasted%20image%2020260409223059.png)
- DB 서버를 분리한 후, 트래픽이 증가하면서 애플리케이션 서버가 병목 지점이 될 수 있음
- Scale Up으로 우선 대응할 수 있으나, Scale Up에 한계가 오는 시점이 있음
- 이때는 Scale Out을 도입해야 하는 시점이며, 서버가 여러 대가 됨에 따라 추가적으로 고려해야 할 사항들이 생김
	- 로컬 캐시의 동기화 문제
	- 애플리케이션 서버 간 트래픽 균등 분산 -> ELB를 통해 적용
- 또한 Scale Out을 도입하면 특정 애플리케이션 서버가 갑자기 다운되어도 서비스를 정상적으로 운영할 수 있어 가용성이 올라감

## 읽기 전용 데이터베이스(Read Replica) 도입
![](imgs/Pasted%20image%2020260409223356.png)
- 다중 데이터베이스는 **안할 수 있으면 안하는 게 좋음**. 이걸 1순위 해결책으로 생각하면 안 됨
- 데이터베이스가 병목이 된다면, 일차적으로 쿼리 튜닝, 인덱스, 역정규화 등으로 성능 개선을 시도해야 함
- DB가 여러 대가 되는 순간 DB 서버 간 데이터 동기화 문제도 발생하고, DB 자체가 매우 비싼 자원이기 때문
- 단일 DB 구조에서 충분히 최적화를 했음에도 정말 트래픽이 많아 DB가 병목이 된다면, 읽기 전용 데이터베이스를 추가하는 것을 고려하는 편이 그나마 나음

## 캐시 서버 도입
![](imgs/Pasted%20image%2020260409223736.png)
- 다중 애플리케이션 서버를 구축하면서부터 로컬 캐시는 불일치 문제로 해결책이 될 수 없을 수도 있음
- 이 경우 별도의 글로벌 캐시 스토리지(ex: Redis)를 두어 캐싱
- 캐싱을 적용하면 데이터를 더 빨리 제공할 수 있음은 물론, DB의 부하를 줄일 수 있다는 이점이 존재

## 병목 지점에 따른 성능 개선 체크리스트
### 애플리케이션 서버 병목
- 애플리케이션 로직에 비효율적인 부분은 없는가?
- 용량이 큰 정적 파일을 애플리케이션 서버에서 제공하고 있는가?
- 단일 서버이고 이미 충분한 리소스를 갖고 있다면, 다중 서버로 분리할 수 있는가?
### 데이터베이스 병목
- 비효율적인 쿼리가 있는가? (복잡한 JOIN, Full Scan이 발생하는 쿼리 등)
- 별도 캐시 스토리지에서 캐싱해서 제공할 수 있는 빈번한 조회가 있는가?
- 최종적 일관성을 허용할 수 있는가? (Read Replica 도입 검토를 위함)
